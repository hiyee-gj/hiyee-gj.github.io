001. Django - Django의 기본 흐름과 tutorial 1 review

기본적으로 django, mysql이 설치되어 있는 상황 이후에 django에 대한
흐름을 이해하기 위한 글입니다

Django (참고영상)

django의 흐름

  1. client가 web application을 보기 위해서 요청
  2. Nginx 또는 Apache가 존재 이 둘은 흔히 상용 단계에서 사용
  3. 장고에서는 개발을 위해서 경량 개발 웹서버를 제공하고 있음
  4. 개발할때는 경량 개발 웹서버를 사용
  5. WSGI는 웹서버와 장고 프레임워크를 연결하기 위해 사용
  6. request가 넘어가고 사용자가 특정 주소를 요청하는데 url 파일에서 요청한 주소를 잘게 나눠줌 이 작업을 parsing이라고 함
  7. 잘게 나눠진 주소들은 역할에 맞게 view로 이동함
  8. view에는 웹 어플리케이션에 각각 맞는 작업을 하기 위한 내가 만든 코드가 존재
    • EX) 요청받은 데이터를 저장, 데이터베이스에서 꺼내오는 데이터베이스 작업, pdf를 생성하는 작업이라던지 등등
  9. view 작업이 끝나면 디자인 작업인 template으로 이동하고 사용자는 response를 받게 된다

프로젝트와 앱의 차이

  • 프로젝트는 mysite와 같은 앱의 묶음
  • 앱은 특정한 기능을 수행하는 웹 어플리케이션
    • EX) 아이스크림 가게를 위한 앱을 만든다고 한다면, 1. 아이스크림 관리를 위한 관리 앱 2. 주문을 위한 주문 앱 3. 이벤트를 위한 이벤트 앱 특정 기능만 수행하는 앱을 각각 만들어서 프로젝트를 구성

django tutorial 1

  • polls라는 앱 생성(투표를 담당하는 앱)
  • view.py를 작성하고 view.py를 호출하기 위해 urls.py를 만들고, 경로?? 입력
  • 최상위 URLconf에 방금 만든 polls의 url을 연결
  • 최상위 URLconf는 mysite의 urls.py에 있음
  • iclude 함수는 다른 url을 참조할 수 있게 해줌

여기까지 흐름

  1. 현재 최상위 URLconf에 내가 만든 polls 앱의 url을 연결함
  2. 예를 들어 127.0.0.1/polls/라는 url이 있다면 이 url을 parsing해서 polls라는 해당 경로를 잡아내고 polls.urls로 연결시켜줌
  3. /polls/ 앱 urls는 전달받고 이후에 아무것도 없기 때문에 path 내에서 view.index라는 view 내부로 연결 시켜줌
  4. view 내부의 index 함수에서는 내가 작성한 "hello world"라는 response를 client에게 전달 해줌
