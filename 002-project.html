002. Project - pagination, path variable, query parameter

Pagination

  • 객체를 한 화면에서 얼마나 보여줄 지 정하는 것을 말한다
  • 이 과정에서 어떤 방식으로 페이지를 나눌지 담당 프론트엔드 개발자와 협의가 필요하다
  • 흔히 offset, limit으로 기능 구현하는데, 이 방식이 좀 더 manual한 방법이라면, django는 이 기능을 지원하는 paginator라는 모듈이 존재한다
  • 하지만, 이번 프로젝트에서는 최대한 장고가 제공하는 편의 기능들에 해당하는 수동적인 메커니즘이 있다면, 공부를 위해 잠시 장고의 모듈은 배제하기로 결심했기 때문에, paginator에 대해서는 구글링을 통해 알아보기 바란다

원리

  • pagination의 원리는 놀랍도록 간단하다
  • 백엔드에서 response할 내용이 담긴 result라는 리스트가 있다고 한다면, pagination은 일정한 기준을 가지고, 이 리스트를 slice 처리하는 것이다
JsonResponse({'RESULT':result[offset:limit]}, status=200)
  • 생각보다 구현은 어렵지 않았다. 다만 offset, limit을 정할 때, 각 페이지에 보여주고 싶은 객체의 수를 정하고, 프론트에서 계산을 통해 query parameter에 offset과 limit을 보내줄지, 백엔드에서 계산을 통해 보내줄지 정해야 하는 부분이 더 고민되는 부분이었다
    • 우리 팀은 query parmeter에 담기는 의미가 좀 더 직관적인 것을 선호 해서 프론트에서 page에 값을 담아 주면, 백엔드에서 계산을 통해 되돌려 주는 방식을 택했다
  • page에 값이 들어 오면, 한 페이지에 보여주고 싶은 객체 수인 page_size를 통해 offset과 limit을 계산해주고, 그 값으로 slice해서 보내주는 방식이다

path variable vs query parameter

Endpoint

  • ENDPOINT란 API가 서버에서 리소스에 접근할 수 있도록 가능하게 하는 URL이다

API

  • API는 프로그램이 기능할 수 있게 명령을 정리하고, 명령을 받으면 응용프로그램과 상호작용하여 요청된 명령에 대한 값을 전달해준다
  • API는 프로그램들이 서로 상호작용하는 것을 도와주는 매개체로 볼 수 있다

API의 역할

  • API는 서버와 데이터베이스에 대한 출입구 역할을 한다
    • 데이터베이스에는 소중한 정보들이 저장되는데, 모든 사람들이 이 데이터베이스에 접근할 수 있으면 보안에 문제가 생길 것이다 API는 이를 방지하기 위해 서버와 데이터베이스에 대한 출입구 역할을 하며, 허용된 사람들에게만 접근성을 부여해 주는 역할을 한다

path vs query

  • Endpoint를 구성하다보면, 같은 맥락인데 보여주고 싶은 내용이 다를 수 있다
예를 들면, https://localhost:8000/users

위에서 users로 주로 표현하는 이유가 뭘까? 답은 바로 users에는 하나의 객체가 아닌 여러 객체들로 이뤄져 있기 때문이다.

  • 상품을 보여주는 커머스 사이트를 예로 들어보자. 모든 상품을 보여주고 싶은 /products를 엔드포인트로 갖는 페이지가 있는 반면, 한 상품의 상세 내용을 보여주고 싶은 경우도 있다
  • 이 경우에 보여주는 방법에 대해서 어떻게 표현할 것인가? 이 표현 방법이 바로 path variable과 query parameter의 방식이 존재하는 것이다

그럼 이 두가지 방식 중 어떤 방식을 어떤 상황에 주로 사용하는지 정리해 보겠다

  • 흔히 두 방식 중 resource가 달라지면, path variable을 사용하고, filter의 경우에는 query parameter를 사용한다고 말한다
  • 원리를 살펴보자면, 우선 어떤 resource를 식별해야하는 상황을 생각해보자
    • 쿼리의 경우, 서버에서는 어느 방식으로 넘어오던지 간에 받아온 resource의 정해진 key로 쿼리를 날리게 된다. Query Parameter는 서버로 데이터가 넘어가고 쿼리를 날리며 해당하는 데이터가 없을 경우 따로 에러 핸들링을 해줘야 하는 번거로움이 존재한다
    • Path Variable은 저 경로의 존재하는 페이지가 없으므로 404 에러를 발생한다

즉 resource를 식별해야하는 상황에서는 Path Variable가 더 적합하다

  • 당연히 filter의 경우 filter 되는 값이 없다고 404를 띄울 수 없기 때문에 query의 방식이 더 유연할 것이다(빈 리스트를 보여줄 것이므로)
