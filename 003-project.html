003. Project - gi & github, products view

Git & Github

  • git과 github에는 많은 기능들이 있고, 더 어려운 기능들도 많다고 한다 이번 프로젝트를 진행하면서, git에 대해서 고통받는 부분이 많아서 따로 설명을 추가적으로 듣게 되었다. 그 내용에 대해서 정리하는 글이다

순서

  • 프로젝트가 시작되고, github에 repo가 생성되었다. main branch를 clone 받아서 초기세팅을 완료했고, push해서 github에도 반영해주었다
  • 초기세팅이 완료되고, 각각 clone을 통해 github에 있는 main과 같은 버전의 main을 내 local에 저장했다
  • 나는 이 main에서 내가 담당하는 기능의 이름을 지닌 feature/기능 branch를 하나 생성했고, 그 안에서 작업을 계속 했다

생겼던 의문

  • 각각 처음은 같은 버전의 main에서 시작했지만, 작업하다보면 각자 속도도 다를 것이기 때문에 먼저 push된 branch가 merge된다면, 내 local의 main과 github의 main이 달라지는 점에서 의문이 들었다
    • 이런 상황을 해결하기 위해(협업을 위해) merge된 새로운 버전의 main이 존재한다면, 나는 작업이 끝난후 내 branch에서 우선, add와 commit 단계만 실시한다(log에는 commit 내용이 보일 것임)
    • 그 상태에서 branch를 main으로 다시 갈아끼운다(내 local에 있던 main임 github의 새로운 main과는 다름)
    • 그럼 내 상태는 main branch이고, 이 상태에서 github에 있는 새로운 버전의 main branch를 pull해와서 버전을 일치시킨다
    • 그 후 다시 내 branch(featur/기능)으로 갈아끼운 후 push한다
  • 여기까지가 기본적인 흐름이고, 아주 기초적인 내용이라고 한다……나중에 더 어려운게 있다고 하니 그 때 다시 한번 정리하도록 하겠다

Products View

  • 처음에는 막막하게 시작했지만, 작성하다보니 모든 기능을 한번에 구현하려는 욕심을 버리는 것이 오히려 좋은 시작단추가 되었다
  • 우선 큰 틀을 닦아놓은 후, 이 mainview에서라면 우선 모든 상품을 보여주는 view를 작성한 후, 조건들을 추가해 나갔다
  • 중간 쯤 보이는 list comprehension은 코드의 길이도 줄여주지만, for문보다 무려 2배가 빠르다고 한다. 우선 내 실력이 되는 부분은 간단한 부분의 list comprehension이었지만, 피드백 후 바꿀 수 있는 부분에 대해선 고민해 보려 한다
  • hover되는 창에 카테고리를 표시해 줄 view이다
  • main에 비해서 어려운 부분이 없었다. 지금 글을 쓰면서 보니, pagination 부분은 필요없는 부분 같아서 제거했다
  • 내게 고통을 선사한 검색 view이다
  • 서로 다른 테이블의 조건들을 한번에 filter해야 하는 부분에서 정말 많이 고민했고, 실패했고, 점점 더 복잡해져가는 찰나 django shell에서 여러 관계를 뒤적거린 후 한숨을 쉬게된 작업이었다
  • 생각보다 쉽게 불어올 수 있는 방법이 있었고, Q라는 모듈의 갓갓 기능도 한 몫 해줬다
  • 이 view부터는 굳이 하나의 값이 들어있는 리스트라면 굳이 list화 된 상태로 reponse하지 않고, 프론트엔드에서 좀 더 포장을 뜯기 쉽게 객체로 불러와서 보내는 여유를 가지고?? 작성한 view이다
